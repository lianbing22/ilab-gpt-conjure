# 产品案例：从通用 WebUI 到品牌图像工作台

## 一句话结论

这次改造的价值不是“增加一个 Logo 开关”，而是将不可控的 AI 生图结果转换为可交付、可追溯、可在手机上完成的企业内容工作流。

## Founder Gate

- **问题不是**：给现有生图 WebUI 换一套移动端样式。
- **真正的问题是**：内容生产者在手机上需跨过长表单才能生成，而将品牌标识交给模型生成又会破坏一致性，最终仍需人工返工。
- **最小验证**：不重写原生客户端，先在现有 Web 工作区中打通“提示词 → 生成 → 确定性品牌合成 → 原图/品牌图交付”，并让手机首屏能看到核心操作。
- **成功证据**：移动端能完成主链路；原图不被覆盖；同一请求可幂等处理；单张品牌合成失败不影响其他结果；自动化测试和部署版本一致。
- **反证信号**：真实用户仍需要导出到其他工具完成品牌处理；手机端完成率无提升；高级设置折叠导致频繁返工；品牌合成成功率不足以替代人工。

## 背景与观察

上游项目已具备图像生成、编辑、历史任务、并发队列和桌面打包能力，但主交互是桌面双栏工作台。移动端早期实现只是将模块纵向堆叠，导致：

- 固定头部、底部生成栏和空预览持续占用可用视口。
- 参考图、品牌素材和输出参数在生成前全部展开，主任务被次要信息切断。
- 品牌 Logo/Slogan 如果写入提示词，会被模型解释、扭曲或重画。
- 品牌后处理引入了新的多线程元数据写入，暴露了原有读-改-写的竞争风险。

## 产品假设

1. 将高频主操作保留在首屏，将低频配置收敛为摘要，能降低手机端首次生成成本。
2. 将品牌标识从生成模型中分离，使用确定性后处理，能在保留原图的同时交付品牌图。
3. 将品牌请求在入队时冻结，并为后处理加入幂等和失败隔离，能让历史任务可追溯、可恢复。

## 方案

### 1. 移动优先信息层级

- 固定头部从 `68px` 压缩到 `52px`。
- 底部操作栏收敛为 `68px + safe-area`，生成按钮高度 `48px`。
- 无结果预览改为 `80px` 状态行，加载、错误或结果出现时恢复完整预览。
- “素材与参考”默认显示选中数量摘要，保留点击和键盘展开。
- 历史任务抽屉收敛头尾，为列表保留主要垂直空间。

详细交互和验收见 [移动端重构](mobile-ux-redesign.md)。

### 2. 确定性品牌合成

- Logo 和 Slogan 作为独立图层，可分别选择和启用。
- 品牌模板冻结素材内容哈希、摆放策略、边距和版本。
- 合成器根据局部背景选择明色/暗色素材，必要时使用可控的 scrim。
- 原始输出永不被覆盖，品牌图文件名嵌入请求哈希前缀。
- 相同请求和原图已成功时直接复用；多张图逐张隔离失败。

### 3. 存储与恢复

品牌后处理和任务队列可由不同线程更新同一份 `metadata.json`。解决方案不是对单个调用加补丁，而是建立统一存储原语：

```text
per-task RLock
    → read latest metadata
    → mutate in memory
    → write sibling temp file
    → flush + fsync
    → os.replace
    → best-effort SQLite index upsert
```

JSON 保持为真实数据源，SQLite 索引失败不会掩盖已成功的任务写入，后续可通过刷新索引修复偏差。

## 结果与证据

- 已部署品牌版 `v0.1.1`：[www.amtr.cloud](https://www.amtr.cloud/)。
- 移动端空状态首屏可同时看到提示词、素材摘要、输出摘要和预览状态。
- 品牌结果与原图分开保存，前端可切换、下载和查看处理状态。
- 品牌合成、任务恢复、并发写入和移动布局都有自动化回归覆盖。
- 当前没有公开的付费、留存或原生应用上架数据，文档不使用这些未验证信号。

## 下一步的最小验证

| 指标 | 定义 | 判断用途 |
| --- | --- | --- |
| 手机端首次生成时长 | 打开页面到首次点击生成 | 验证信息压缩是否减少操作成本 |
| 移动端完成率 | 手机端完成任务 / 发起任务 | 判断是否值得继续投入移动交互 |
| 品牌合成成功率 | 成功品牌图 / 成功原图 | 验证后处理能否稳定替代人工 |
| 失败恢复率 | 重试后成功的任务 / 发起重试的失败任务 | 判断恢复机制是否真正有用 |

## 面试可讲的 90 秒版本

> 我接手的是一个成熟开源生图 WebUI，而不是从空项目开始。我先定义两个产品问题：桌面表单在手机上阻断生成，以及 AI 会随机重画企业 Logo。我将移动端重构为提示词、摘要、生成和结果优先的工作流，同时把 Logo/Slogan 从生成模型中分离，改成生成后确定性合成。为了让它可交付，我保留不可变原图，加入幂等、逐图失败隔离和崩溃恢复。这又暴露了原有任务元数据的并发丢失风险，我继续修到任务级锁和原子写入，并补齐回归测试。产品现在以 v0.1.1 部署，但我不会将 Web/PWA 说成已上架的原生 App；下一步先用完成率和合成成功率验证是否值得继续投入。
